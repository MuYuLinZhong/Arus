## 3. 路由与页面对应

### 3.1 路由总表

| 路径 | 名称 | 组件 | 鉴权 | 角色 | 功能概要 |
|------|------|------|------|------|----------|
| `/login` | Login | Login.vue | 否 | — | 手机号+密码登录；已登录则重定向至 /dashboard |
| `/` | 布局 | Layout.vue | 是 | — | 侧栏+顶栏+内容区，子路由渲染在内容区 |
| `/`（默认） | — | — | 是 | — | redirect → /dashboard |
| `/dashboard` | Dashboard | Dashboard.vue | 是 | admin | 总览：统计卡片、设备状态分布、最新告警 |
| `/users` | Users | Users.vue | 是 | admin | 用户管理：列表、筛选、新建/编辑/重置密码/启用禁用 |
| `/devices` | Devices | Devices.vue | 是 | admin | 锁具管理：列表、筛选、添加锁具 |
| `/permissions` | Permissions | Permissions.vue | 是 | admin | 权限管理：列表、授权、撤销 |
| `/audit-logs` | AuditLogs | AuditLogs.vue | 是 | admin | 审计日志：筛选、游标分页加载更多 |
| `/alerts` | Alerts | Alerts.vue | 是 | admin | 告警管理：列表、处置 |
| `/403` | Forbidden | Forbidden.vue | 否 | — | 无权限提示，按钮返回首页 |
| `/:pathMatch(.*)*` | — | — | — | — | redirect → /dashboard |

### 3.2 路由守卫逻辑

- 访问 `/login` 且已登录 → 重定向到 `/dashboard`。
- 除 `/login`、`/403` 外，`meta.requiresAuth !== false` 且未登录 → 重定向到 `/login`。
- 若路由带 `meta.roles`，当前用户 `role` 不在列表中 → 重定向到 `/403`。

## 5. 状态管理与数据流

### 5.1 全局 Store（Pinia）

| Store | 文件 | 状态 | 方法 | 持久化 | 说明 |
|-------|------|------|------|--------|------|
| auth | stores/auth.ts | token, userUUID, role, name | setAuth, clearAuth | localStorage 同步 | 登录态；请求头与路由守卫依赖 |
| alert | stores/alert.ts | pendingAlerts, pendingCount | fetchPendingAlerts | 无 | 待处置告警；Layout 轮询与角标展示 |

### 5.2 本地状态（页面内）

| 页面 | 主要状态 | 说明 |
|------|----------|------|
| Login | form, loading | 表单与提交中状态 |
| Dashboard | dashboard, chartRef | 接口数据与图表 DOM |
| Users | users, loading, page, total, searchText, filterRole/filterStatus, showCreateModal/showEditModal, createForm/editForm, creating/updating | 列表、分页、筛选、弹窗与表单 |
| Devices | devices, loading, page, total, searchText, filterStatus, showCreateModal, createForm, creating | 同上 |
| Permissions | permissions, loading, page, total, filterStatus, showGrantModal, grantForm, granting | 列表、分页、筛选、授权弹窗 |
| AuditLogs | logs, loading, hasMore, cursor, filterDeviceID, filterAction, dateRange | 游标分页、筛选、加载更多 |
| Alerts | alerts, loading, page, total, filterStatus/filterSeverity/filterDeviceID, showHandleModal, currentAlert, handleForm, handling | 列表、分页、筛选、处置弹窗 |

### 5.3 数据流概要

- 登录：Login 调用 `login()` → 写入 `authStore.setAuth(result)` → 跳转 `/dashboard`。
- 登出：Layout 调用 `logout()` 后 `authStore.clearAuth()` → 跳转 `/login`。
- 列表数据：各页 onMounted 或筛选/分页变更时调用对应 `api/admin` 或 `api/lock`，结果写入本地 ref。
- 告警角标：Layout 通过 `useAlertPolling` 定时调用 `alertStore.fetchPendingAlerts()`，侧栏与顶栏展示 `pendingCount`。

## 6. API 调用与后端对应

### 6.1 请求封装（utils/request.ts）

- 实例：`baseURL: '/api'`，`timeout: 10000`。
- 请求：从 localStorage 取 token 写入 `Authorization: Bearer <token>`，并设置 `X-Request-ID`。
- 响应：`data.code === 0` 时返回 `data.data`；否则根据 code 提示并 reject。
- 业务码 1001/1003：清登录、跳转登录、提示「会话已过期」。
- 业务码 1002：清登录、跳转登录、提示「账号已被禁用」。
- 2xxx：提示「权限不足」。
- 其它：`message.error(data.message || '请求失败')`。
- 网络/HTTP：401 清登录并跳转登录；400/429/5xx 及超时、网络错误均有对应 message。

### 6.2 认证（api/auth.ts）

| 方法 | 后端路径 | 说明 |
|------|----------|------|
| login(data) | POST /api/auth/login | 登录，返回 token、user_uuid、role、name、expires_at |
| logout() | POST /api/auth/logout | 退出（需 Token） |

### 6.3 锁具（api/lock.ts）

| 方法 | 后端路径 | 说明 |
|------|----------|------|
| getAuthorizedDevices() | GET /api/lock/devices | 当前用户授权设备列表 |
| submitChallenge(data) | POST /api/lock/challenge | 挑战请求 |
| reportResult(data) | POST /api/lock/report | 上报开锁结果 |

### 6.4 管理后台（api/admin.ts）

| 方法 | 后端路径 | 说明 |
|------|----------|------|
| getDashboard() | GET /api/admin/dashboard | 总览统计与最近告警 |
| getUsers(params) | GET /api/admin/users | 用户列表（分页、筛选） |
| createUser(data) | POST /api/admin/users | 新建用户 |
| updateUser(uuid, data) | PUT /api/admin/users/:uuid | 更新用户 |
| resetPassword(uuid) | POST /api/admin/users/:uuid/reset-pwd | 重置密码 |
| getDevices(params) | GET /api/admin/devices | 设备列表 |
| createDevice(data) | POST /api/admin/devices | 添加锁具 |
| getPermissions(params) | GET /api/admin/permissions | 权限列表 |
| grantPermission(data) | POST /api/admin/permissions | 授权 |
| batchGrantPermissions(data) | POST /api/admin/permissions/batch | 批量授权 |
| revokePermission(id) | DELETE /api/admin/permissions/:id | 撤销授权 |
| getAuditLogs(params) | GET /api/admin/audit-logs | 审计日志（游标分页） |
| getAlerts(params) | GET /api/admin/alerts | 告警列表 |
| handleAlert(id, data) | PUT /api/admin/alerts/:id | 处置告警 |

**未在本前端调用的后端接口**（供移动端或其它端使用）：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/health` | 健康检查，CI/CD 或运维使用；前端可选用于「系统状态」展示 |

| POST | `/api/lock/report` | 上报开锁结果，移动端开锁后上报 |

## 7. 统一响应与错误码

与 [服务端 §13 统一响应与错误码](../server/01-服务端总体设计.md#13-统一响应与错误码) 对齐，前端按以下约定处理。

### 7.1 响应信封

所有接口（除 `/metrics`）返回统一结构：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "request_id": "uuid",
  "timestamp": 1708300000000
}
```

- **code === 0**：成功，业务数据在 `data` 中；请求封装（utils/request.ts）直接返回 `data.data` 给调用方。
- **code !== 0**：业务错误，`data` 为 `null`，`message` 为可展示的说明；由响应拦截器统一弹 message 并按码做跳转/清登录。

### 7.2 业务错误码与前端行为

| 范围 | 含义 | 前端处理 |
|------|------|----------|
| **1001** | 登录失败（用户不存在或密码错误） | 拦截器 message 提示，不跳转（当前在登录页） |
| **1002** | 账号禁用 | 清登录态、跳转 `/login`、message「账号已被禁用」 |
| **1003** | 会话过期 / Token 无效 | 清登录态、跳转 `/login`、message「会话已过期，请重新登录」 |
| **2xxx** | 权限不足（2001 无操作权限、2002 访问拒绝） | message「权限不足」；若在路由守卫后出现可跳转 `/403` |
| **3xxx** | 锁具（3001 设备不存在、3002 设备不可用、3003 请求过频） | message 展示 `data.message`；3003 对应后端 429 |
| **4xxx** | 参数（4001 参数错误、4002 请求过期） | message 展示 `data.message` |
| **5xxx** | 服务内部错误 | message 展示 `data.message` 或「请求失败」 |

### 7.3 HTTP 状态码与前端行为

| HTTP | 场景 | 前端处理 |
|------|------|----------|
| 200 | 成功，body 中 code 可能非 0 | 以 body.code 为准，见 7.2 |
| 401 | 未提供 Token 或服务端未返回 body | 清登录态、跳转 `/login` |
| 403 | 禁止访问 | message 提示，可选跳转 `/403` |
| 429 | 限流（全局/登录/挑战） | message「请求过于频繁，请稍后再试」 |
| 4xx/5xx、超时、网络错误 | 请求失败 | message 提示，不统一清登录（除 401） |

### 7.4 request_id

响应头或 body 中的 `request_id` 可用于与后端日志关联排查问题；前端可在控制台或错误上报时携带，当前未强制展示给用户。

## 8. 安全与敏感数据

与 [服务端 §11 安全设计](../server/01-服务端总体设计.md#11-安全设计) 配合，前端需遵守以下约定。

### 8.1 Token 携带方式

- 所有需鉴权接口：请求头 `Authorization: Bearer <token>`。
- Token 来源：登录成功后由 `authStore` 持久化到 localStorage，请求拦截器从 localStorage 读取并写入 Header。
- 不在 URL、Cookie 中传递 Token。

### 8.2 CORS 与同源

- 后端配置 CORS，允许前端所在源访问 `/api`。
- 开发环境通过 Vite 代理将 `/api` 转发至后端，避免跨域；生产环境前端与后端同域或后端配置允许的前端域名。

### 8.3 敏感数据展示与存储

| 数据 | 后端策略 | 前端策略 |
|------|----------|----------|
| **手机号** | 列表接口可脱敏（如 138****8888） | 直接展示接口返回字段，不做二次脱敏；若后端未脱敏可前端 MaskPhone |
| **密码** | 仅 Argon2 哈希存储，任何接口不返回 | 前端不展示、不存储明文密码；创建用户后展示「初始密码已生成」及重置密码返回的临时密码（由管理员告知用户） |
| **Token** | 登录接口返回一次 | 仅存 localStorage，不打印、不写入日志；登出或 1002/1003 时清除 |

### 8.4 重置密码返回值

- 后端 `POST /api/admin/users/:uuid/reset-pwd` 在响应 `data` 中返回 `new_password`（明文，一次性）。
- 前端应在处置成功后展示或复制给管理员，由管理员通过安全渠道告知用户；不写入持久化存储。

## 9. 与后端接口契约

与 [服务端 §6 路由与接口设计](../server/01-服务端总体设计.md#6-路由与接口设计) 保持一致，避免前后端参数不一致。

### 9.1 分页

| 场景 | 请求参数 | 响应字段 | 说明 |
|------|----------|----------|------|
| **偏移分页** | `page`（从 1 起）、`page_size` | `items`、`total` | 用户/设备/权限/告警列表 |
| **游标分页** | `cursor`（可选）、`limit` | `items`、`next_cursor`、`has_more` | 审计日志，按 occurred_at DESC |

### 9.2 列表筛选参数

与后端接口文档一致，前端传参与后端 Query 或 Body 约定一致：

| 接口 | 常用筛选参数 | 备注 |
|------|--------------|------|
| GET /api/admin/users | role, status, search, page, page_size | search 为姓名模糊 |
| GET /api/admin/devices | status, pipeline_tag, search, page, page_size | search 为 device_id/name 模糊 |
| GET /api/admin/permissions | user_id, device_id, status, page, page_size | device_id 为字符串 |
| GET /api/admin/audit-logs | user_id, device_id, action, start_time, end_time, cursor, limit | 时间范围 Unix 毫秒或 ISO8601 视后端约定 |
| GET /api/admin/alerts | status, device_id, severity, page, page_size | — |

### 9.3 管理操作请求体

- 创建用户：phone, name, department, role 等，与后端 CreateUser 请求体一致。
- 创建设备：device_id, name, location, longitude, latitude, pipeline_tag, risk_level, device_key（hex）等。
- 授权：user_id, device_id, device_type（默认 "lock"）, valid_from, valid_until。
- 处置告警：handle_note；consecutive_fail 类型可传 unlock_device: true。

具体字段名与类型以服务端文档与当前实现为准，前端 types 与 api 层应与后端保持一致。

| **告警管理** | 列表（分页+筛选）、处置（备注+可选解除设备锁定） | §9.4 告警管理 |
| **错误与限流** | 统一错误码提示、1002/1003 清登录并跳转、429 限流提示 | §13 错误码、§5 限流 |

### 10.2 未在本前端实现（后端已提供）

| 能力 | 说明 |
|------|------|
| 健康检查 | GET /api/health，可供运维或前端「系统状态」页 future 使用 |
| 锁具端开锁流程 | GET /api/lock/devices、POST /api/lock/challenge、POST /api/lock/report 由移动端或其它客户端使用，本管理后台不包含开锁 UI |

### 10.3 与后端行为对齐的注意点

- **密码**：创建用户后初始密码、重置密码后新密码均由后端生成并在响应中返回，前端不通过短信发送，由管理员当面或安全渠道告知用户（与服务端 §9.1 一致）。
- **权限模型**：设备以 `device_type` + `device_id` 标识，授权/列表接口均带 device_type（默认 "lock"），与数据库设计一致。
- **告警处置**：仅 `consecutive_fail` 类型支持处置时勾选「解除设备告警锁定」，对应后端解除 devices_lock.status 锁定。

## 11. 交互模式

### 11.1 登录与登出

| 场景 | 行为 |
|------|------|
| 登录 | 手机号、密码必填；提交后 loading，成功则 setAuth、message.success、跳转 /dashboard；失败由 request 拦截器统一 message |
| 登出 | 顶栏用户下拉「退出登录」→ 调 logout()，无论成功失败均 clearAuth、message.success、跳转 /login |
| 未登录访问 | 路由守卫跳转 /login |
| 已登录访问 /login | 路由守卫跳转 /dashboard |

### 11.2 导航

| 方式 | 实现 |
|------|------|
| 侧栏 | Layout 内 a-menu，点击 `$router.push` 至各业务路径；选中态由 route.path 同步 selectedKeys |
| 面包屑 | 顶栏展示当前 route.meta.title，默认「总览」 |
| 用户菜单 | 顶栏 a-dropdown，展示头像+姓名，「退出登录」 |
| 告警入口 | 顶栏铃铛 + 侧栏「告警管理」旁 Badge，点击进入 /alerts |

### 11.3 表单

| 场景 | 组件 | 校验与提交 |
|------|------|------------|
| 登录 | a-form + a-input / a-input-password | 前端非空校验；提交后 loading，成功/失败见 7.1 |
| 新建用户 | a-modal 内 a-form | 手机号、姓名、部门、角色；确认 loading，成功关弹窗、清表单、刷新列表 |
| 编辑用户 | a-modal 内 a-form | 姓名、部门、角色；确认 loading，成功关弹窗、刷新列表 |
| 添加锁具 | a-modal 内 a-form | 设备ID、名称、位置、经纬度、管线、风险等级、设备密钥；同上 |
| 授权 | a-modal 内 a-form | 用户 ID、设备 ID、生效/失效时间；确认后 grantPermission、刷新列表 |
| 处置告警 | a-modal 内 a-form | 处置备注、可选「同时解除设备告警锁定」；仅 consecutive_fail 显示解锁选项 |

### 11.4 列表与分页

| 页面 | 分页方式 | 筛选 |
|------|----------|------|
| Users | 偏移分页（page、pageSize、total），onChange 拉新页 | 搜索、角色、状态 |
| Devices | 同上 | 搜索、状态 |
| Permissions | 同上 | 状态 |
| Alerts | 同上 | 状态、级别、设备ID |
| AuditLogs | 游标分页（cursor、limit、has_more），「加载更多」追加 | 设备ID、操作类型、时间范围，查询时重置列表 |

### 11.5 弹窗与二次确认

| 场景 | 实现 |
|------|------|
| 新建/编辑/授权/处置 | a-modal，v-model:open 控制显隐，@ok 提交，:confirm-loading 防重复 |
| 重置密码 | a-popconfirm，确认后调用 resetPassword，success 提示（新密码由后端在响应中返回，前端可按需展示或由管理员另行告知用户） |
| 启用/禁用用户 | a-popconfirm，确认后 updateUser(status)、刷新列表 |
| 撤销授权 | a-popconfirm，确认后 revokePermission、刷新列表 |

### 11.6 错误与反馈

| 类型 | 处理 |
|------|------|
| 接口业务错误 | request 响应拦截器：根据 code 弹 message、部分清登录并跳转 |
| 网络/HTTP 错误 | 401/400/429/5xx 及超时、网络异常均有 message 提示 |
| 成功操作 | message.success（如「更新成功」「告警已处置」） |
| 列表加载 | 表格 :loading，部分页有「加载更多」按钮 loading |
